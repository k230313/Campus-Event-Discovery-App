What the Comparison Reveals
1. Table-Stakes Features Are Universal

Across all seven university platforms analysed, a consistent baseline of functionality exists. Every portal includes a dedicated events page, detailed event descriptions (including time, location, and overview), and a mechanism for RSVP or registration.
These features are considered table-stakes, meaning they are expected by default and no longer provide any competitive advantage.
This uniformity indicates that universities are aligned on the minimum functional requirements for event delivery, but it also highlights a lack of innovation. Since all platforms meet these baseline expectations, users have little reason to prefer one system over another based on functionality alone. As a result, differentiation must come from enhanced user experience, intelligence, or integration, rather than core features.
2. Filtering Is Universally Weak

While some level of filtering exists across platforms, it is generally limited and inconsistent. The University of Queensland (UQ) stands out slightly by offering both date and category filtering, but even this is only a partial solution. The remaining institutions provide only single-dimension filtering (e.g., by date or category), or very limited refinement options.
Critically, no platform enables multi-dimensional filtering-such as combining date, location, category, audience type, and keyword search simultaneously. This creates friction for users who are trying to quickly identify relevant events from a large pool.
The implication is that current systems are designed more for browsing than discovery. Students must manually sift through irrelevant events, increasing cognitive load and reducing engagement. This represents a major opportunity for improvement through advanced filtering and search capabilities.
3. Personalisation Is Completely Absent

One of the most significant findings is the total absence of personalised recommendations across all platforms. None of the seven systems analysed leverage user data-such as past attendance, interests, course enrolment, or behaviour-to suggest relevant events.
This is a major gap when compared to modern digital platforms (e.g., streaming services or e-commerce), where personalisation is a core expectation. Without it, students are presented with generic event listings that may not align with their preferences or schedules.
The absence of personalisation suggests that current platforms operate on a one-size-fits-all model, which is inefficient and outdated. Introducing recommendation systems could dramatically improve engagement by surfacing highly relevant, timely, and tailored events to each user.
4. Social Features Do Not Exist

- None of the analysed platforms incorporate social or peer-driven elements such as:

Visibility of which friends or classmates are attending
Trending or popular events
Attendance counts or engagement metrics
Reviews, ratings, or comments
This lack of social proof is significant because student behaviour is often influenced by peer participation and community trends.
Without these signals, events lack context and perceived value, making it harder for users to decide what is worth attending.
In contrast, modern platforms frequently use social validation to drive engagement. The absence of these features suggests that university event systems are designed as static information boards, rather than interactive, community-driven ecosystems.

5. Mobile Apps Are Essentially Missing

Despite the prevalence of mobile-first usage among students, dedicated mobile applications for event discovery are largely absent. For example, Macquarie University's EventsAir app requires an event-specific code, meaning it functions as a tool for accessing pre-registered events rather than discovering new ones.
All other platforms rely on responsive web design accessed via browsers. While functional, these are often less intuitive, slower, and less engaging than purpose-built mobile applications.
This indicates a failure to align with user behaviour trends, where students expect seamless, app-based experiences with features like push notifications, calendar integration, and real-time updates. The lack of mobile optimisation further reduces accessibility and ongoing engagement.
6. Fragmentation Is the Defining Problem

Perhaps the most critical issue identified is the fragmentation of event information across multiple platforms within each university. Typically, institutions operate 2-4 separate event portals, including:

Central university events

Faculty or department-specific events

Student union or club events

Student life or extracurricular platforms

These systems operate independently, forcing students to search across multiple sources to get a complete view of available events. This decentralisation leads to missed opportunities, duplicated effort, and user frustration.

Fragmentation reflects an organisational silo problem, where different departments manage their own systems without integration. From a user perspective, this creates a disjointed experience and undermines the goal of effective event discovery.

A unified platform would address this by consolidating all event data into a single, centralised ecosystem, significantly improving usability and engagement.

SECTION 3

Market Opportunity
Why the Problem Remains Unsolved
These seven portals represent institutions with significant digital investment, yet the core event discovery problem identified in the project brief persists across all of them. Three structural reasons explain why:

Reason 1 - Portals are built for administrators, not students

University event portals are built and maintained by Communications or IT teams to publish events, not to help students discover them. The primary user in the design process is the event organiser publishing content - not the student trying to find something to do on a Thursday afternoon. This results in listing tools, not discovery tools.

Reason 2 - Club and union events are structurally separated

University IT systems cannot easily publish club/society content because clubs are governed by independent student unions (Arc UNSW, USU, UQU, YouX). This creates a permanent structural gap between institutional events and the social/extracurricular events students actually care most about. No portal has solved this integration problem.

Reason 3 - Personalisation requires user data infrastructure that universities don't build

Delivering personalised recommendations requires user accounts, interest profiles, behavioural data, and recommendation logic. Universities don't build this for event portals because it's outside their core competency and requires ongoing engineering investment. This is precisely where a purpose-built third-party app has a structural advantage.

Alignment with Project Scope
The project brief provided by Skillup Labs outlines a set of core features that align directly with the gaps identified in the university portal analysis. Each feature is not arbitrary-it is a targeted solution to a specific market failure:

1. Event Listings (Create & Manage)

This feature directly addresses the absence of club and society events from official university portals. In the current landscape, many student-led events exist in isolated systems (e.g., club pages, social media), making them difficult to discover.

By enabling organisers to self-publish events into a centralised platform, the system removes institutional bottlenecks and ensures broader visibility. This transforms event management from a top-down controlled process into a more inclusive, user-driven ecosystem.

2. Event Discovery (Browse Upcoming)

All existing portals rely heavily on static lists that require users to manually scroll and search for relevant events. This creates a passive and inefficient discovery experience.

A dynamic discovery feed replaces this with a continuously updating stream of relevant events, shifting the experience from searching to surfacing. This aligns with modern content platforms where users expect personalised, real-time discovery rather than static browsing.

3. Advanced Filtering (Category, Date, Interest)

This feature directly solves one of the most consistent weaknesses identified: limited filtering functionality. While platforms like the University of Queensland offer partial filtering, no system supports multi-dimensional filtering.

By allowing users to combine filters such as category, date, and personal interests, the platform significantly reduces information overload. This enables precision discovery, where users can quickly narrow down events that are most relevant to them.

4. Keyword Search

Only a minority of existing portals provide any meaningful search functionality, and those that do often lack full-text capability.

Introducing robust keyword search establishes a baseline usability feature that most current systems fail to deliver. It allows users to directly locate events using natural language queries, making the platform more intuitive and efficient.

5. Bookmarking / Saving Events

Current platforms lack persistent saving mechanisms, with limited exceptions such as temporary features during orientation periods (e.g., partial implementation at the University of Sydney).

A year-round bookmarking system allows users to curate their own event lists, improving planning and engagement. This shifts the platform from a one-time browsing tool into a continuous engagement system that supports user intent over time.

6. Personalised Dashboard

No analysed platform offers personalised recommendations, making this the single most valuable differentiator in the proposed solution.

A personalised dashboard leverages user behaviour, preferences, and saved data to surface relevant events automatically. This transforms the platform into an intelligent system rather than a static directory, significantly enhancing user experience and engagement.

7. User Accounts

User accounts act as the foundation layer that enables several advanced features, including personalisation, saved preferences, and bookmarking.

Without accounts, the system cannot retain user data or adapt over time. By introducing authentication and user profiles, the platform becomes capable of learning from user behaviour, enabling continuous improvement in event discovery.

Recommended Stack Rationale

The proposed technology stack-React, Node.js, and a relational database-is well-aligned with both the project requirements and the gaps identified in existing systems:

1. React Frontend

Using React enables the development of a highly dynamic and responsive user interface.
Unlike the static, page-reload-based systems used by current university portals, React supports:
Real-time filtering and updates
Seamless navigation without full page reloads
A more engaging, app-like user experience
This is critical for implementing features such as dynamic event feeds and instant search results, directly addressing usability limitations in existing platforms.
2. Node.js Backend / REST API

Node.js provides a scalable backend environment capable of handling event data, user interactions, and API requests efficiently.
A RESTful API architecture allows:
Separation of frontend and backend concerns
Easy integration with future mobile applications
Scalable handling of users, events, and interactions
This makes the system flexible and future-proof, supporting both current prototype needs and potential expansion.
3. Relational Database

A relational database (e.g., PostgreSQL or MySQL) is particularly well-suited for this application due to the structured relationships between entities:
Users
Events
Categories
Bookmarks
RSVPs
These relationships require efficient querying and joining, especially for features like filtering and recommendations. A relational model ensures data integrity, consistency, and performance.
4. Cloud Deployment (Heroku / AWS)

Deploying the application on platforms such as Heroku or Amazon Web Services ensures the prototype is publicly accessible.
This is critical for:
Real-world user testing
Gathering feedback from students
Demonstrating a working product to stakeholders
Cloud deployment also provides scalability, reliability, and ease of maintenance, which are essential for transitioning from prototype to production.
Overall Strategic Fit

The chosen stack is not only technically appropriate but also strategically aligned with the project goals. It enables the development of a platform that is:
Dynamic rather than static
User-centric rather than system-centric
Scalable and extensible for future growth
Together, these technologies provide the foundation needed to deliver a solution that meaningfully improves upon the current fragmented and limited event discovery
