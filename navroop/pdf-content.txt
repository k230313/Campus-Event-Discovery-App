1. Introduction
1.1 Purpose
The purpose of this Software Requirements Specification (SRS) document is to
provide detailed information about the Campus Event Discovery App. The SRS
document will include all details regarding the functionalities and non-functionalities of
the system, its design, and architecture. This SRS will be used by developers and
stakeholders during the development process.
1.2 Scope
The Campus Event Discovery App is an online platform designed to consolidate
access to campus events. Students will have access to browsing, searching, and saving
events on the platform. Organizers will have the opportunity to create and manage their
events on the platform. The platform aims at increasing visibility and participation from
students in campus events.
1.3 Project Objectives
The project objectives are as follows:
● Centralized discovery of campus events
● Ability to browse and search events
● Ability to save and track events
● Ability to create and manage events
● Intuitive and friendly user interface
1.4 Problem Description
At most learning institutions, information regarding events can be found in
different places including emails, social media, and hard copies of posters. This leads to
the inability of the learners to know all the activities taking place within the institution,
which results in most of the events lacking attendance due to unawareness of their
happening. There should be a platform that would combine all the event information.
1.5 Methodology
The project will follow the Software Development Lifecycle (SDLC) using
an agile approach. The development process will include discovery, requirement
analysis, system design, development, testing, and deployment. Iterative
development will be used to continuously improve the system based on feedback
and testing results.

2. Business and System Overview
2.1 Business Description
Campus Event Discovery App is an EdTech web-based tool that focuses on increasing
student involvement by implementing an effective event discovery system. Academic
workshops, social events, career events and club activities are regular activities in campuses,
yet students do not attend such events because of low visibility and dissemination of
information.
Students are now informed of events in different channels including emails, posters, social
media and university internal portals. This has however resulted in confusion and reduced
accessibility by students.
This system is intended to:
Improve the rapport between organizers and students of the events.
Enhance student participation in activities organized in the university.
Offer an effective event discovery system to students.
This system will serve to serve as a liaison between organizers of the event and the students
and enable students to easily access information.
2.2 Current System (Existing Process)
The existing event discovery system is via a manual system. The information is shared in
different mediums including:
Emailing students.
Social media such as Facebook and Instagram.
Physical posters and notice boards.
University internal portals
Weaknesses of the existing System:
Lacks of centralized event discovery.
Communication is done through different channels, which confuses.
Students are missing out on various events
None of the information filtering options.
No options for personalization
No opportunities to monitor the interest of students by the organizers.
As a result of these drawbacks, students tend to be overwhelmed with information and
students are sometimes not aware of what has taken place leading to low student
engagement.

2.3 Proposed System
The system is going to be a full-stack web-based event management system at the campus.
The proposed system has the following features:
Event organizers creating and managing events.
Browse and filtering of events by the student community.
Search facility
Bookmarking/saving events
Personalized dashboard
The system will facilitate the following features to the students and event organizers:
Students will have the option of finding events that they are interested in with ease.
The event organizers will be able to easily publish the events.
This system is bound to overcome the shortcomings of the current system since it will offer
one system.
2.4 System Overview
The Campus Event Discovery App will be based on a three-tier architecture system.
1. Presentation Layer (Frontend)
React.js will be used to develop this layer.
2. Application Layer (Backend)
This layer will be built with the help of Node.js/Express.
3. Data Layer (Database)
MySQL or MongoDB will be used to develop this layer.
The system will have the following functions:
● Authentication (register and log in)
● List and filter events.
● Find and add bookmarks.
● Personalized dashboard
The system will be hosted on a cloud platform.

● RAM: 4 GB minimum (8 GB recommended)

● Network: Stable internet connection with a static local IP address or

All three layers of the application are hosted on a single machine; the React frontend
static files, the Node.js back-end process, and the MySQL database. Nginx runs on the
server to receive incoming HTTP traffic, serve the frontend build and proxy API
requests to the Node.js process running on port 3001. PM2 is used to control the
Node.js process which is automatically restarted in case of crash or reboot.
This is a single-server configuration that is suitable to a prototype that has a small
number of simultaneous users. Within a production setting, a load balancer would be
suggested to separate the database onto a special host and introduce a load balancer to
spread traffic between multiple application server instances.

5.3 Database Design
A database design of the Campus Event Discovery App is based on MySQL relational
database with four basic entities: users, events, bookmarks, and categories. The schema
will aid the three main functions of the system, which include user authentication, event
management, and bookmarking.
Account credentials and role information are stored in the users table, where there is a
distinction between Students and Organisers. Events table contains all the event listing
information that Organisers have generated such as event title, description, date, time,
location, category and publication status. The bookmarks table is a join table that solves
the many-to-many relationship between Students and Events, and it is where the
relationship between what students have saved and what events is recorded. The
categories table is a basic reference table to categorize the events and aid in filtering.
Referential integrity is ensured by the use of foreign key relationships: every event is
referenced by its creating organiser through the organiser_id, every event is categorised
by a category through the category_id and every bookmark is associated between a
student and an event through the student-event relationship. The passwords are never
kept in plain text form, only the values of bcrypt-hashes are stored in the users table.
The database automatically records all timestamps.
The entire schema is captured in three sub-sections, an Entity Relationship Diagram
(5.3.1), the entire table definitions, including data type and constraints (5.3.2), and a
field-level Data Dictionary (5.3.3).

6. Security and Data Protection
6.1 Data Encryption
The system will ensure that user data is protected during transmission and
storage.
● All communication between the frontend and backend will use HTTPS and
TLS/SSL to secure data in transit and for encryption respectively.
● User passwords will be stored using bcrypt hashing, which includes the in-built
salting to enhance resistance for bruteforce method of third party access and the
password will never be stored in plain text.
● Authentication will be handled using JSON Web Tokens (JWT), which will expire
after a defined period to improve security.
● Sensitive configuration data (such as secret keys, database connection, etc) will
be stored using environment variables.
6.2 Data Anonymization
The system will protect user privacy by limiting exposure of personal data.
● User email and sensitive information will not be publicly visible.
● Only necessary information such as organiser name will be shown in event
details. For example, in event listings, only the organiser’s name and event
title/details will be visible while all other user specific information remains
hidden
● User activity such as saved/bookmarked events will remain private to each user
and accessible only to the individual user. This privacy is maintained consistently
across all parts of the system including dashboards, and reports.

member as a server) instead of a paid cloud service (AWS, Azure, or Google Cloud). This
choice will remove the expenses of continued hosting and satisfy the prototype
demonstration needs of the project.
9.2.1 Rational for Self-hosting
The following reasons have been selected to use self-hosting:
• 	Cost - no subscription costs on cloud compute, storage or bandwidth.
• 	Control - complete access to the server environment, which makes it easier
to configure and debug.
• 	Scope alignment - the project deliverable is a prototype, not a production
scale system.
• 	Learning value - students will have practical experience in Linux server
administration, Nginx set up, and process management.

Explain Difference

Diagram
Section 5 was supplemented with a written
explanation of what the ERD and the class diagram
are. The data storage structure is modeled by the
ERD (Section 5.3.1) - it specifies tables, primary and
foreign keys and the cardinality of the relationships
between the entities in the stored data. The class
diagram is a model of the software design - it
specifies the classes, their attributes and methods
and the manner in which objects interact at the
application logic level. They are similar in their
names of entities; however, they have different
purposes: the ERD tells how to design a database
schema, and the class diagram tells how to
implement objects in an object-oriented way.

13. Conclusion
The Campus Event Discovery App has been developed as an all-inclusive solution to
the problems that relate to disjointed event information in educational
institutions. As we have identified in the problem statement, students have
always been weak in terms of keeping up with what is taking place within the
campus due to lack of a centralized platform. The system is an effective solution
to that issue as it provides a single and easy-to-use application, which makes it
possible to discover, manage, and interact with different events.
The system has been carefully defined in this Software Requirements Specification
(SRS), in terms of business requirements, functional and non-functional
requirements, system architecture and design components. The three-layer
architecture proposed will also render it scalable, maintainable, and there will be
a clear separation of concerns among the frontend, backend, and the database. In
addition, new technologies, such as React.js, Node.js, and MySQL are employed
and contribute to the development of a responsive and efficient web application.
Not only does this system enhance the amount of engagement of the students since
the user is able to browse, search and bookmark events, but it also provides the
event organisers with tools to create, maintain and monitor the performance of
events. The mechanisms that ensure JWT authentication, password hashing
using bcrypt and secure data transmission protocols have been incorporated to
ensure security and protection of data, thereby ensuring privacy of the user and
reliability of the system.
Further, the project planning aspects like Work Breakdown Structure (WBS),
milestones, and Gantt chart are a well-organized and planned plan of
development switching to the principles of Agile SDLC. These factors make the
project viable, controllable, and meet the set schedule.
In conclusion, Campus Event Discovery App is a scalable and efficient solution to
improving campus communication and engagement. The system will also offer a
very strong foundation to the new developments such as supporting mobile
applications, advanced analytics, and integrating with third-party platforms. All
in all, this project shows that software engineering principles can be effectively
used to address a real-life issue.
